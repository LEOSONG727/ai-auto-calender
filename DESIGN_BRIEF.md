# AI Auto Calendar Design Brief

작성일: 2026-06-03
문서 목적: AI Auto Calendar의 실제 UI/UX 디자인 기준을 정의한다.

## 1. 디자인 한 줄 목표

사용자가 생각나는 일을 아무렇게나 적으면, 제품이 조용히 정리하고, 가능한 시간에 배치하고, 사용자가 안심하고 승인할 수 있게 만드는 시간 운영 인터페이스.

이 제품은 캘린더 앱처럼 보여도 안 되고, 할 일 앱처럼 보여도 안 된다. 사용자가 느껴야 하는 첫인상은 "내 시간을 대신 챙겨주는 비서가 생겼다"이다.

## 2. UX 핵심 판단

### 2.1 사용자는 정리하고 싶어서 들어오지 않는다

사용자는 정리된 사람이 되고 싶지만, 정리하는 행위 자체는 귀찮다. 따라서 첫 행동은 반드시 "입력"이어야 한다.

나쁜 첫 화면:

- 비어 있는 캘린더
- 복잡한 대시보드
- 설정 먼저 요구
- 긴 온보딩 설문
- AI 기능 설명

좋은 첫 화면:

- 지금 바로 적을 수 있는 입력창
- 오늘 시간표
- 승인 대기 중인 제안
- 위험한 일정/마감만 조용히 표시

### 2.2 사용자는 AI보다 자기 캘린더를 더 신뢰한다

AI 제안은 절대 확정처럼 보이면 안 된다. 제안은 제안이고, 확정은 사용자의 승인이다.

화면에서 구분해야 하는 것:

- 기존 캘린더 일정: 이미 확정된 시간
- AI 제안: 아직 캘린더에 반영되지 않은 후보
- AI 타임블록: 사용자가 승인한 계획
- 위험/충돌: 사용자가 판단해야 할 문제

### 2.3 개인화는 설명보다 체감이다

"AI가 학습합니다"라고 말하지 않아도 된다. 대신 사용자는 이런 문장을 봐야 한다.

- "문서 작업은 보통 90분 걸려서 넉넉히 잡았어요."
- "회의 직후 작업은 자주 미뤄져서 30분 뒤로 배치했어요."
- "오전 집중 시간이 가장 성공률이 높아 이 시간에 추천했어요."

개인화는 기능 설명이 아니라 추천 이유 안에 자연스럽게 나타나야 한다.

## 3. UX 원칙

### 3.1 입력은 1초 안에 시작되어야 한다

사용자가 앱을 열고 입력하기까지의 거리가 가장 중요하다.

- 로그인 후 첫 화면 상단 또는 하단에 항상 Quick Capture를 둔다.
- placeholder는 기능 설명이 아니라 실제 입력을 유도한다.
- 모바일에서는 엄지로 바로 누를 수 있는 하단 입력 버튼을 둔다.
- 입력 중에는 필드명을 요구하지 않는다.

추천 placeholder:

```text
생각나는 일정이나 할 일을 그대로 적어보세요
```

### 3.2 AI 정제 결과는 수정하기 쉬워야 한다

AI가 틀릴 수 있다는 전제로 디자인한다.

정제 결과 카드에서 바로 수정 가능해야 하는 값:

- 제목
- 유형: 일정/할 일/반복/메모
- 마감
- 예상 소요시간
- 우선순위
- 희망 시간

수정 UI는 설정폼처럼 무겁지 않아야 한다. chip, dropdown, inline edit, stepper를 사용한다.

### 3.3 캘린더는 보기보다 판단을 돕는 도구여야 한다

캘린더 UI의 목적은 예쁜 시간표가 아니다. 사용자가 "이 제안이 말이 되는지" 빠르게 판단하도록 돕는 것이다.

필수 표현:

- 기존 일정
- 빈 시간
- AI 제안
- 승인된 타임블록
- 충돌
- 보호 시간
- 마감 임박 작업

### 3.4 승인과 수정은 같은 무게여야 한다

승인만 강조하면 사용자는 AI를 따라야 한다고 느낀다. 수정이 쉬워야 사용자가 통제감을 갖는다.

제안 카드의 기본 액션:

- 승인
- 시간 변경
- 오늘 못함
- 삭제

보조 피드백:

- 이 시간 좋아요
- 너무 짧아요
- 너무 길어요
- 회의 직후는 피해주세요

### 3.5 화면은 조용해야 한다

이 제품은 사용자의 바쁜 하루를 다룬다. 화면이 들뜨면 안 된다.

피해야 할 것:

- 과한 그라디언트
- 큰 AI 캐릭터
- 마케팅 문구
- 장식용 카드
- 설명이 많은 empty state
- 매번 뜨는 모달

필요한 것:

- 높은 가독성
- 명확한 시간축
- 빠른 스캔
- 안정적인 컬러 체계
- 작은 피드백
- 신뢰감 있는 여백

## 4. 정보 구조

### 4.1 최상위 내비게이션

권장 탭:

1. Today
2. Plan
3. Inbox
4. Profile

데스크톱:

- 좌측 사이드바
- 중앙 Today/Plan
- 우측 AI 제안 패널

모바일:

- 하단 탭
- Quick Capture 플로팅 버튼 또는 하단 고정 입력
- AI 제안은 bottom sheet

### 4.2 Today

목적:

사용자가 오늘 무엇을 언제 해야 하는지 바로 이해한다.

구성:

- Quick Capture
- 현재 시간 기준 Next Action
- 오늘 시간표
- 승인 대기 제안
- 마감 위험
- 미완료/재배치 필요 항목

Today 화면에서 사용자가 답해야 하는 질문:

- 지금 바로 해야 할 일이 무엇인가?
- 오늘 시간이 부족한가?
- AI 제안을 승인해야 하는가?
- 미뤄진 일이 있는가?

### 4.3 Plan

목적:

이번 주 시간을 운영한다.

구성:

- 주간 캘린더 그리드
- 기존 일정
- AI 타임블록
- 제안 중인 블록
- 드래그로 시간 이동
- 우선순위/마감 필터

Plan 화면에서 중요한 것은 "조정"이다. 사용자는 AI가 짠 계획을 보고 직접 바꾸고 싶어 한다.

### 4.4 Inbox

목적:

아직 정리되지 않은 입력을 처리한다.

구성:

- AI 정제 대기
- 질문 필요
- 일정 후보
- 할 일 후보
- 삭제/보류

Inbox는 부담스러운 할 일 더미가 되면 안 된다. "정리되지 않은 생각을 제품이 대신 들고 있는 곳"이어야 한다.

### 4.5 Profile

목적:

개인화와 권한을 관리한다.

구성:

- 업무 시간
- 보호 시간
- 집중 시간
- 회의 후 버퍼
- 자동화 수준
- 연결된 캘린더
- 데이터 삭제/연결 해제

Profile은 설정 화면이지만, 사용자는 자주 들어오지 않아야 한다. 대부분의 개인화는 행동으로 학습되어야 한다.

## 5. 핵심 화면 상세

### 5.1 Today 화면

데스크톱 레이아웃:

```text
┌────────────────────────────────────────────────────────────┐
│ Sidebar │ Main Today Timeline              │ AI Panel       │
│         │ Quick Capture                    │ Pending        │
│         │ Now / Next                       │ Risks          │
│         │ Day schedule                     │ Learnings      │
└────────────────────────────────────────────────────────────┘
```

모바일 레이아웃:

```text
┌─────────────────────────┐
│ Today header            │
│ Quick Capture           │
│ Now / Next              │
│ Timeline                │
│ Pending suggestions     │
│ Bottom tabs             │
└─────────────────────────┘
```

중요한 컴포넌트:

- 현재 시간 라인
- 다음 행동 카드
- 확정 일정 블록
- AI 타임블록
- 승인 대기 카드
- 위험 알림

### 5.2 Quick Capture

상태:

- idle
- typing
- saved
- parsing
- parsed
- needs question
- failed

입력 후 흐름:

1. 사용자가 텍스트 입력
2. 즉시 저장
3. 작은 상태 표시: "정리 중"
4. AI 정제 결과가 카드로 나타남
5. 사용자는 승인하거나 수정

중요:

- 입력 저장과 AI 처리 상태를 분리한다.
- AI 처리가 늦어도 사용자가 불안하지 않아야 한다.
- 실패해도 원문은 남아야 한다.

### 5.3 AI Suggestion Card

카드 구성:

- 정제된 제목
- 유형 배지
- 추천 시간
- 예상 소요시간
- 우선순위
- 추천 이유
- 개인화 근거
- 대안 시간
- 승인/수정/미루기/삭제

추천 이유 예시:

```text
금요일 마감이고 90분 집중 시간이 필요해요. 내일 오전은 회의가 많아서 오늘 16:00에 배치했어요.
```

개인화 근거 예시:

```text
문서 작업은 보통 예상보다 오래 걸려 90분으로 잡았어요.
```

### 5.4 Weekly Plan

주간 계획은 캘린더 앱과 비슷하지만, 목적이 다르다.

기존 캘린더:

- 확정된 약속 확인

AI Auto Calendar:

- 가능한 시간에 일을 배치하고 조정

필수 인터랙션:

- 블록 드래그 이동
- 블록 길이 조정
- 제안 승인
- 충돌 블록 표시
- 미완료 블록 재배치

### 5.5 Learning Feedback

피드백은 별도 설문이 아니라 자연스러운 행동이어야 한다.

피드백 트리거:

- 사용자가 제안을 수정했을 때
- 사용자가 작업을 완료했을 때
- 사용자가 미뤘을 때
- 일정이 반복적으로 실패했을 때

피드백 문구:

- "이 시간대는 앞으로 피할까요?"
- "비슷한 작업은 90분으로 잡을까요?"
- "회의 직후에는 버퍼를 더 둘까요?"

버튼:

- 네
- 이번만
- 아니요

## 6. Visual Direction

### 6.1 전체 인상

조용한 업무 도구, 개인 비서, 시간 운영실.

키워드:

- calm
- precise
- trustworthy
- focused
- lightweight
- adaptive

### 6.2 컬러

권장:

- 배경: 따뜻하지도 차갑지도 않은 neutral
- 텍스트: 고대비 charcoal
- 확정 일정: muted blue/gray
- AI 제안: soft green 또는 teal 계열
- 위험: restrained red
- 보호 시간: subtle hatch 또는 pale gray
- 완료: muted green

피해야 할 것:

- 보라색 AI 서비스 느낌
- 강한 네온
- 하나의 색으로만 구성된 단조로운 팔레트
- 어두운 대시보드 스타일

### 6.3 타이포그래피

원칙:

- 숫자와 시간이 잘 보여야 한다.
- 제목은 짧고 명확해야 한다.
- 설명 문장은 1-2줄을 넘지 않는다.
- 버튼 텍스트는 행동 중심이어야 한다.

권장 텍스트 계층:

- Page title
- Section title
- Time label
- Task title
- Reason text
- Metadata

### 6.4 아이콘

아이콘은 설명을 줄이는 데만 사용한다.

권장:

- Plus: 입력 추가
- Check: 승인/완료
- Clock: 시간/소요시간
- Calendar: 일정
- Alert: 충돌/위험
- Refresh: 재배치
- Sliders: 선호 설정

## 7. Interaction Rules

### 7.1 승인

승인 버튼은 명확해야 하지만 과하게 크지 않아야 한다. 사용자가 실수로 누르지 않게 secondary action과 분리한다.

승인 후:

- 카드가 사라지지 않고 "캘린더에 추가됨" 상태로 짧게 남는다.
- 되돌리기 버튼을 제공한다.

### 7.2 수정

수정은 모달보다 inline 또는 bottom sheet를 우선한다.

수정 가능한 값:

- 시간
- 길이
- 우선순위
- 마감
- 제목

### 7.3 미루기

"미루기"는 부끄러운 행동처럼 보이면 안 된다. 바쁜 사람에게 미루기는 정상적인 일정 운영이다.

미루기 옵션:

- 오늘 나중에
- 내일
- 이번 주 안에
- 직접 선택

### 7.4 완료

완료는 개인화의 핵심 데이터다.

완료 후 가벼운 질문:

- "예상 시간과 비슷했나요?"
- "다음에도 이 시간대가 좋나요?"

단, 매번 묻지 않는다. 반복 패턴을 보정해야 할 때만 묻는다.

## 8. Empty / Loading / Error States

### 8.1 Empty Today

나쁜 문구:

```text
아직 일정이 없습니다. 일정을 추가해보세요.
```

좋은 문구:

```text
오늘은 비어 있어요. 생각나는 일을 적으면 가능한 시간에 배치해드릴게요.
```

### 8.2 AI Parsing Loading

```text
입력은 저장했어요. 일정 형식으로 정리 중입니다.
```

### 8.3 Calendar Sync Error

```text
캘린더를 불러오지 못했어요. 입력한 내용은 저장되어 있고, 연결이 복구되면 다시 배치할게요.
```

### 8.4 Low Confidence

```text
이건 마감이 있는 할 일인가요, 실제 약속인가요?
```

## 9. Accessibility

필수:

- 시간 블록 색상만으로 구분하지 않는다.
- 키보드로 제안 승인/수정 가능해야 한다.
- 캘린더 블록에는 텍스트 라벨을 둔다.
- 작은 시간 라벨 대비를 확보한다.
- 모바일에서 터치 타겟은 최소 44px 이상.
- 긴 제목은 줄바꿈 또는 말줄임 기준을 명확히 한다.

## 10. Responsive Strategy

데스크톱:

- 계획 조정과 주간 캘린더에 최적화
- 우측 패널로 AI 제안 상시 표시
- 드래그 앤 드롭 지원

모바일:

- 빠른 입력과 오늘 확인에 최적화
- 제안 상세는 bottom sheet
- 주간 캘린더는 과하게 복잡하게 만들지 않는다
- 오늘/내일 중심으로 보여준다

태블릿:

- 데스크톱 구조를 단순화
- Today + AI Panel 2열 우선

## 11. Design QA Checklist

디자인 결과는 다음 질문으로 검토한다.

- 앱을 열자마자 입력할 수 있는가?
- AI 제안과 확정 일정이 명확히 구분되는가?
- 사용자가 수정하기 쉬운가?
- 제안 이유가 납득되는가?
- 개인화가 부담스럽지 않게 드러나는가?
- 캘린더가 복잡해 보이지 않는가?
- 모바일에서 엄지로 주요 액션을 누를 수 있는가?
- AI가 캘린더를 마음대로 건드린다는 불안이 없는가?
- 에러가 나도 원문 입력이 보존된다는 느낌이 있는가?
- 하루가 실제로 정리된다는 감각이 있는가?

## 12. 구현 핸드오프 기준

디자인 산출물은 다음을 포함해야 구현 가능하다.

- 데스크톱 Today
- 모바일 Today
- Quick Capture 상태별 UI
- AI Suggestion Card 상태별 UI
- Weekly Plan
- Inbox
- Preferences
- Empty/loading/error states
- 디자인 토큰: colors, typography, spacing, radius
- 컴포넌트 상태: default, hover, selected, disabled, error

## 13. 최종 디자인 방향

AI Auto Calendar의 디자인은 사용자를 설득하는 디자인이 아니라, 사용자의 바쁜 하루를 받아주는 디자인이어야 한다.

제품은 이렇게 말해야 한다.

```text
일단 적어두세요.
제가 정리하고, 가능한 시간을 찾아보고, 당신이 승인할 수 있게 보여드릴게요.
쓸수록 당신에게 더 맞춰질게요.
```
