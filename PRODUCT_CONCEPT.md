# AI Auto Calendar Product Concept

작성일: 2026-06-02
리포지터리: `LEOSONG727/ai-auto-calender`
제품 표시명: AI Auto Calendar

## 1. 제품 한 줄 정의

AI Auto Calendar는 사용자가 떠오르는 일정과 할 일을 편하게 적으면, 기존 캘린더와 개인 성향을 바탕으로 실제 가능한 시간대에 타임블록을 제안하고, 승인된 계획을 캘린더에 반영하는 AI 비서형 스케줄 오퍼레이터다.

이 제품의 본질은 "캘린더 앱"이 아니라 "바쁜 사람의 시간을 대신 운영해주는 비서"다. 사용자는 입력만 하고, 제품은 정리, 판단, 배치, 조정, 설명을 맡는다.

## 2. 고객 문제

사람들은 일정과 할 일을 다른 방식으로 기억한다.

- 확정된 약속은 캘린더에 들어간다.
- 해야 할 일은 머릿속, 메모장, 카톡 나에게 보내기, Google Keep, Notion, 종이 메모에 흩어진다.
- 바쁜 날에는 할 일이 있어도 "언제 할지"까지 정하지 못한다.
- 일정 변경이 생기면 하루 전체 계획을 다시 짜야 한다.
- 중요한 일이 급한 일에 밀리고, 작은 일은 계속 미뤄진다.
- 캘린더에는 회의만 있고, 실제 일할 시간은 보이지 않는다.

사용자가 진짜 원하는 것은 할 일 목록이 아니다. "내가 오늘 무엇을 언제 하면 되는지"를 믿을 수 있게 정해주는 것이다.

## 3. 타겟 사용자

### 3.1 1차 타겟

일정과 업무가 동시에 많은 지식 노동자.

- 스타트업 창업자, 리더, PM, 디자이너, 개발자
- 미팅과 실무가 섞여 있는 회사원
- 여러 프로젝트를 동시에 관리하는 프리랜서
- 개인 일정, 업무 일정, 자기계발 일정을 함께 관리하려는 사용자

### 3.2 2차 타겟

비서가 필요한데 실제 비서를 둘 수 없는 사람.

- 임원 후보군, 팀 리드, 1인 사업자
- 반복적으로 일정 조율과 우선순위 판단을 해야 하는 사용자
- ADHD 성향, 잦은 망각, 일정 관리 피로가 있는 사용자

### 3.3 초기에는 제외할 타겟

- 여러 명의 회의 시간을 자동 조율하는 팀 스케줄링 제품
- 법무, 의료, 금융처럼 일정 실수가 치명적인 고위험 전문직 전용 제품
- 완전 자동 캘린더 운영을 원하는 엔터프라이즈 관리자

## 4. 핵심 가치 제안

### 4.1 고객이 체감해야 하는 변화

사용자는 이렇게 느껴야 한다.

> "생각나는 대로 적었는데, 내 하루가 정리됐다."

제품이 제공해야 하는 가치는 다음 네 가지다.

- 기억 부담 감소: 떠오르는 일을 놓치지 않는다.
- 판단 부담 감소: 우선순위와 예상 소요시간을 AI가 1차로 정리한다.
- 배치 부담 감소: 실제 가능한 시간에 할 일을 넣어준다.
- 조정 부담 감소: 일정이 바뀌어도 다시 짜준다.

### 4.2 차별화

기존 캘린더 앱은 "확정된 시간"을 보여준다.
기존 할 일 앱은 "해야 할 항목"을 보여준다.

AI Auto Calendar는 "지금 있는 시간 안에서 무엇을 언제 해야 하는지"를 운영한다.

## 5. 제품 원칙

### 5.1 사용자는 대충 입력할 수 있어야 한다

사용자에게 처음부터 제목, 날짜, 시간, 반복, 우선순위, 소요시간을 요구하면 실패한다. 제품은 사용자가 막 적은 문장을 구조화해야 한다.

예:

- "금요일까지 투자자 자료 정리"
- "엄마 병원 예약해야 함"
- "이번 주 운동 3번"
- "내일 2시 민수랑 커피"
- "세금 관련 서류 확인 30분 정도"

### 5.2 AI는 결정자가 아니라 비서다

초기 버전에서 AI는 캘린더를 마음대로 바꾸면 안 된다. 사용자가 승인하기 전까지는 제안 상태로 남겨야 한다.

AI가 해야 할 일:

- 입력 정제
- 일정/할일 구분
- 우선순위 추론
- 시간 배치 제안
- 애매한 점 질문
- 배치 이유 설명

사용자가 해야 할 일:

- 승인
- 거절
- 수정
- 선호 학습 피드백 제공

### 5.3 캘린더를 망치지 않아야 한다

신뢰가 깨지면 제품은 끝난다.

- 기본값은 읽기 중심이어야 한다.
- 쓰기 작업은 사용자 승인 후 진행한다.
- 자동 생성 이벤트는 별도 캘린더에 넣는다.
- 모든 AI 생성 이벤트에는 원본 입력과 생성 이유를 남긴다.
- 되돌리기 기능이 있어야 한다.

### 5.4 설명 가능한 스케줄이어야 한다

사용자는 "AI가 왜 이 시간에 넣었는지" 알아야 한다.

나쁜 설명:

- "AI가 최적의 시간으로 판단했습니다."

좋은 설명:

- "내일 오전은 회의가 연속되어 있어 집중 작업이 어렵습니다. 제안서 초안은 90분 집중 시간이 필요하므로 오늘 16:00-17:30에 배치했습니다."

### 5.5 자동화보다 통제감이 먼저다

초기 사용자는 AI에게 캘린더를 맡기는 데 불안함을 느낀다. 따라서 제품 경험은 다음 순서로 발전해야 한다.

1. 제안만 하기
2. 승인하면 반영하기
3. 낮은 위험의 일정만 자동 반영하기
4. 사용자가 신뢰한 패턴만 자동화하기

### 5.6 쓸수록 개인화되어야 한다

이 제품은 처음부터 완벽한 스케줄을 맞히는 제품이 아니다. 좋은 제품 경험은 "처음부터 100% 정확함"이 아니라 "매주 더 나아짐"이다.

따라서 MVP 단계부터 다음 네 가지 데이터를 분리해서 저장해야 한다.

- 사용자가 원한 것: 원문, 사용자가 직접 고른 우선순위, 희망 시간, 마감, 선호
- AI가 이해한 것: 분류, 예상 소요시간, AI 우선순위, 에너지 요구, 확신도
- 시스템이 제안한 것: 추천 시작/끝 시간, 대안 시간, 배치 이유, 점수
- 사용자가 실제로 한 것: 승인, 수정, 거절, 미룸, 완료, 실제 소요시간

이 네 가지의 차이가 개인화의 원천이다.

예:

- 사용자가 오전 집중 시간을 선호한다고 말했지만 실제 완료율은 오후가 높을 수 있다.
- AI가 30분으로 예측한 일이 매번 70분 걸릴 수 있다.
- 사용자가 "높음"으로 표시한 일도 실제로는 마감 전날에만 처리할 수 있다.
- 회의 직후 15분 안에 넣은 작업은 반복적으로 미뤄질 수 있다.

제품은 이런 차이를 조용히 학습하고 다음 제안에 반영해야 한다. 사용자는 복잡한 설정을 하지 않아도 "얘가 점점 나를 안다"고 느껴야 한다.

## 6. 핵심 사용자 시나리오

### 6.1 빠른 기록

사용자가 이동 중에 앱을 열고 입력한다.

입력:

> "다음 주 화요일까지 랜딩페이지 문구 수정하고 대표님한테 공유"

AI 정제 결과:

- 유형: 할 일
- 제목: 랜딩페이지 문구 수정 및 대표 공유
- 마감: 다음 주 화요일
- 예상 소요시간: 90분
- 우선순위: 높음
- 관련 인물: 대표님
- 필요한 확인: 공유 방식이 이메일인지 메신저인지 불명확

제품 제안:

- 월요일 10:00-11:30 집중 작업
- 화요일 09:30-10:00 최종 검토 및 공유

### 6.2 기존 일정 기반 타임블록

사용자가 Google Calendar를 연결한다.

제품은 다음을 확인한다.

- 이미 확정된 회의
- 바쁜 시간대
- 업무 시간
- 개인 일정
- 집중 가능한 빈 시간
- 점심/저녁/이동/휴식 버퍼

그 후 이번 주 할 일을 배치한다.

### 6.3 일정 변경 후 재배치

갑자기 오후 회의가 생긴다.

제품은 충돌된 타임블록을 감지하고 제안한다.

- "14:00-15:30 제안서 작성 시간이 새 회의와 충돌합니다."
- "오늘 17:00-18:30 또는 내일 09:00-10:30으로 옮길 수 있습니다."
- "마감이 내일 오전이라 내일 09:00보다 오늘 17:00을 추천합니다."

### 6.4 매일 아침 브리핑

아침에 사용자는 오늘 계획을 본다.

화면 구성:

- 오늘의 확정 일정
- 오늘의 AI 타임블록
- 오늘 꼭 끝내야 할 일
- 미뤄도 되는 일
- 위험 알림: "마감 전 확보된 시간이 부족합니다"

### 6.5 점차 개인화되는 경험

첫 주에는 제품이 조심스럽게 묻는다.

- "보통 집중 작업은 오전이 편한가요, 오후가 편한가요?"
- "이 작업은 실제로 몇 분 정도 걸렸나요?"
- "이 시간대가 괜찮았나요?"

둘째 주부터는 행동을 보고 제안이 바뀐다.

- "지난주에 회의 직후 작업은 3번 중 2번 미뤘습니다. 오늘은 회의 후 30분 버퍼를 두겠습니다."
- "문서 작성 작업은 예상보다 평균 35분 더 걸렸습니다. 이번에는 90분으로 잡겠습니다."
- "오전 10시대 집중 작업 완료율이 가장 높아 중요한 작업을 이 시간에 우선 배치했습니다."

사용자는 별도 설정을 많이 하지 않아도 된다. 승인, 수정, 미루기, 완료만 해도 제품은 학습한다.

## 7. MVP 범위

### 7.1 반드시 포함

MVP는 "입력 -> 정제 -> 시간 배치 제안 -> 승인 -> 캘린더 반영"의 루프가 작동해야 한다.

필수 기능:

- Google 로그인
- Google Calendar 읽기
- 별도 `AI Auto Calendar` 캘린더 생성 또는 선택
- 빠른 자연어 입력
- AI 기반 입력 정제
- 할 일과 일정 구분
- 예상 소요시간 추론
- 우선순위 추론
- 사용자가 직접 우선순위/소요시간을 수정할 수 있는 입력
- 업무 가능 시간 설정
- 빈 시간 탐색
- 타임블록 제안
- 승인 후 Google Calendar 이벤트 생성
- 생성된 이벤트 취소/되돌리기
- 오늘/이번 주 계획 보기
- 승인/수정/미루기/완료 행동 로그 저장
- 실제 시작/완료/소요시간을 기록할 수 있는 가벼운 피드백

### 7.2 MVP에서 제외

초기 완성도를 위해 다음은 제외한다.

- 팀원 간 회의 시간 자동 조율
- 다중 캘린더 제공자 완전 지원
- Google Keep 직접 동기화
- 이메일 자동 분석
- 결제
- 모바일 네이티브 앱
- 완전 자동 일정 변경
- 음성 입력

### 7.3 MVP 성공 기준

초기 사용자가 다음 행동을 반복하면 성공이다.

- 하루 3개 이상 빠른 입력을 남긴다.
- AI 정제 결과를 큰 수정 없이 승인한다.
- 제안된 타임블록 중 50% 이상을 승인한다.
- 생성된 타임블록을 실제로 완료 체크한다.
- 다음 날 다시 접속한다.
- 사용자가 수정한 우선순위/소요시간/시간대가 다음 제안에 반영된다.

## 8. 제품 구조

### 8.1 추천 플랫폼

초기 구현은 웹앱 + PWA가 적합하다.

이유:

- Google OAuth와 캘린더 API 연동을 빠르게 검증할 수 있다.
- 주간/일간 타임블록 UI를 구현하기 쉽다.
- 데스크톱 업무 사용자에게 자연스럽다.
- 모바일에서도 빠른 입력 화면을 제공할 수 있다.
- 네이티브 앱 대비 출시와 수정 속도가 빠르다.

단, UX는 모바일 앱처럼 설계해야 한다. 특히 빠른 입력은 첫 화면에서 1초 안에 시작되어야 한다.

### 8.2 향후 앱 전략

MVP 검증 후 네이티브 앱이 필요한 순간:

- 잠금화면/위젯 입력
- 푸시 알림 정교화
- 음성 입력
- 오프라인 캡처
- OS 캘린더와 더 깊은 연동

초기에는 PWA로 충분하고, 실제 사용 패턴이 확인된 뒤 모바일 앱을 만든다.

## 9. 정보 구조

### 9.1 주요 화면

1. Today
   - 오늘 일정과 AI 타임블록
   - 지금 해야 할 일
   - 충돌/위험 알림

2. Capture
   - Google Keep 같은 빠른 입력
   - 여러 줄 자유 입력
   - 입력 즉시 AI 정제 결과 표시

3. Plan
   - 이번 주 타임블록
   - 승인 대기 제안
   - 충돌된 블록
   - 드래그 앤 드롭 수정

4. Inbox
   - 아직 정리되지 않은 캡처
   - 애매해서 질문이 필요한 항목
   - AI가 분류한 할 일/일정 후보

5. Preferences
   - 업무 시간
   - 집중 시간 선호
   - 점심/저녁/휴식
   - 블록 최소/최대 길이
   - 주말 사용 여부
   - 자동화 수준

6. Integrations
   - Google Calendar 연결
   - 캘린더 선택
   - 읽기/쓰기 권한 상태

### 9.2 첫 화면 원칙

첫 화면은 마케팅 페이지가 아니라 실제 작업 화면이어야 한다.

로그인 후 첫 화면:

- 상단: 빠른 입력창
- 중앙: 오늘의 시간표
- 우측 또는 하단: AI 제안/위험/다음 액션

사용자가 제품을 열자마자 "내 시간을 운영해주는 도구"라고 느껴야 한다.

## 10. 상세 기능 정의

### 10.1 Quick Capture

목표:

사용자가 어떤 형식으로든 일정을 기록할 수 있게 한다.

입력 형태:

- 단문
- 여러 줄
- 체크리스트 비슷한 입력
- 날짜가 없는 할 일
- 날짜가 있는 일정
- 반복 습관
- 마감이 있는 업무

기본 동작:

- 입력 후 즉시 저장
- AI 정제는 비동기로 실행
- 사용자는 정제 완료를 기다리지 않아도 된다
- 정제 결과는 Inbox에 쌓인다

필드:

- 원문
- 정제된 제목
- 유형: `event`, `task`, `habit`, `reminder`, `note`, `unknown`
- 마감
- 희망 시간
- 실제 시작/종료 시간
- 예상 소요시간
- 사용자가 직접 선택한 우선순위
- AI가 추론한 우선순위
- 최종 우선순위
- 에너지 레벨
- 위치
- 관련 인물
- 반복 여부
- 확신도
- 필요한 질문
- 사용자가 수정한 항목
- 이전 유사 작업과의 차이

### 10.2 AI Parser

AI Parser는 자연어 입력을 구조화한다.

입력 예:

> "목요일 전에 계약서 검토하고 법무팀에 코멘트 보내기. 오래 걸리진 않을 듯"

출력 예:

```json
{
  "type": "task",
  "title": "계약서 검토 및 법무팀 코멘트 전달",
  "deadline": "2026-06-04T23:59:00+09:00",
  "estimatedMinutes": 45,
  "aiPriority": "high",
  "userPriority": null,
  "finalPriority": "high",
  "energy": "medium",
  "people": ["법무팀"],
  "confidence": 0.78,
  "questions": [
    "검토할 계약서 위치나 링크가 있나요?"
  ]
}
```

확신도 정책:

- 0.85 이상: 바로 제안 생성
- 0.65 이상: 제안 생성하되 수정 가능 상태 강조
- 0.65 미만: 사용자에게 질문 먼저

### 10.3 Calendar Sync

Google Calendar 연동은 MVP의 핵심이다.

공식 Google Calendar API 기준:

- 이벤트 조회: Events list
- 이벤트 생성: Events insert
- 바쁜 시간 조회: FreeBusy query
- 권한은 필요한 최소 scope를 선택해야 한다

MVP 권장 방식:

- 읽기: 사용자의 주요 캘린더 이벤트 조회
- 쓰기: 사용자가 선택한 별도 캘린더에만 AI 생성 이벤트 작성
- 동기화: 주기적 pull sync 우선, 이후 push notification 추가

주의:

- 사용자의 기본 캘린더에 바로 쓰지 않는다.
- AI가 생성한 이벤트는 `extendedProperties.private`에 내부 ID를 저장한다.
- 원본 사용자 일정과 AI 타임블록을 명확히 구분한다.

### 10.4 Scheduling Engine

스케줄링은 LLM에 전부 맡기지 않는다. LLM은 해석과 설명을 담당하고, 실제 시간 배치는 규칙 기반 엔진이 계산한다.

입력:

- 기존 캘린더 이벤트
- 사용자의 업무 가능 시간
- 할 일 목록
- 마감
- 예상 소요시간
- 우선순위
- 에너지 요구
- 선호 시간대
- 버퍼 정책
- 사용자가 완료/미룬 이력
- 사용자가 직접 수정한 우선순위와 소요시간
- 유사 작업의 실제 소요시간
- 시간대별 승인률과 완료율
- 회의 전후 작업 성공률

출력:

- 추천 타임블록
- 대안 시간대
- 배치 이유
- 충돌 위험
- 사용자가 답해야 할 질문
- 개인화 반영 근거

배치 규칙:

- 확정 일정과 충돌하지 않는다.
- 최소 15분 단위로 배치한다.
- 집중 작업은 60분 이상 빈 시간에 우선 배치한다.
- 회의 직후 10-15분 버퍼를 둔다.
- 점심/저녁/수면 시간은 기본적으로 보호한다.
- 마감이 가까운 일은 더 앞에 둔다.
- 낮은 에너지 작업은 회의 사이 짧은 빈 시간에 배치한다.
- 사용자가 자주 미루는 시간대는 점수를 낮춘다.
- 사용자가 자주 완료하는 시간대는 점수를 높인다.
- AI 예상 소요시간보다 사용자의 실제 평균 소요시간을 우선한다.
- 사용자가 직접 입력한 우선순위는 AI 추론보다 우선한다.
- 같은 유형의 작업이 반복적으로 실패한 배치 패턴은 피한다.

스코어링 예시:

```text
score =
  deadlineUrgency * 0.24 +
  finalPriorityWeight * 0.20 +
  focusFit * 0.16 +
  energyFit * 0.10 +
  calendarBufferFit * 0.10 +
  explicitPreferenceFit * 0.08 +
  historicalCompletionFit * 0.08 +
  predictedDurationFit * 0.04
```

초기에는 이 점수를 규칙 기반으로 계산한다. 데이터가 쌓이면 `historicalCompletionFit`, `predictedDurationFit`부터 개인화 모델로 대체한다.

### 10.5 Approval Flow

AI 제안은 승인 전까지 캘린더에 반영하지 않는다.

제안 카드 구성:

- 제목
- 추천 시간
- 예상 소요시간
- 우선순위
- 이유
- 개인화 근거
- 대안 시간
- 버튼: 승인, 수정, 미루기, 삭제
- 빠른 피드백: 시간이 좋음, 너무 이른 시간, 너무 짧음, 너무 김

승인 후:

- Google Calendar에 이벤트 생성
- 내부 DB에 이벤트 ID 저장
- 생성 로그 저장
- 되돌리기 가능
- 제안과 실제 사용자 선택의 차이 저장

### 10.6 Rescheduling

재배치는 이 제품의 품질을 좌우한다.

재배치 트리거:

- 새 일정 추가
- 기존 일정 변경
- 사용자가 할 일을 완료하지 않음
- 마감 임박
- 사용자가 "오늘 못함" 선택
- 사용자가 새로운 우선순위 입력

재배치 원칙:

- 이미 시작한 블록은 바꾸지 않는다.
- 오늘 남은 시간 안에서 먼저 해결한다.
- 마감이 낮은 일은 내일 이후로 미룬다.
- 중요한 일은 작은 블록으로 쪼개서라도 배치한다.
- 변경 전후를 사용자에게 보여준다.

## 11. 개인화 학습 시스템

이 제품의 장기 경쟁력은 캘린더 연동이나 자연어 입력이 아니다. 같은 입력을 받아도 사용자마다 다른 시간표를 제안할 수 있는 개인화 능력이다.

단, 초기부터 사용자별 모델을 파인튜닝하는 방식은 적합하지 않다. 데이터가 부족하고, 비용이 높고, 설명 가능성이 떨어진다. MVP와 초기 고도화 단계에서는 다음 구조가 현실적이다.

- LLM: 자연어 입력 해석, 질문 생성, 배치 이유 설명
- 개인화 프로필: 사용자별 선호와 행동 패턴 저장
- 스케줄링 엔진: 개인화 프로필을 점수 계산에 반영
- 학습 파이프라인: 실제 행동 로그를 요약해 개인화 프로필 업데이트

### 11.1 학습 원칙

개인화는 사용자의 시간을 더 뺏으면 안 된다.

좋은 학습 UX:

- 사용자가 승인/수정/미루기/완료를 누르는 것만으로도 학습된다.
- 질문은 꼭 필요한 순간에만 한다.
- 설정 화면에 너무 많은 선택지를 넣지 않는다.
- 사용자가 왜 제안이 바뀌었는지 이해할 수 있어야 한다.

나쁜 학습 UX:

- 처음부터 긴 설문을 요구한다.
- 매 작업마다 평가를 요구한다.
- "AI 학습을 위해 피드백을 주세요" 같은 부담을 준다.
- 사용자가 원하지 않는 자동화를 갑자기 시작한다.

### 11.2 온보딩에서 받을 최소 데이터

초기 온보딩은 5분이 아니라 30초여야 한다.

필수:

- 기본 업무 시간
- 보호할 시간: 식사, 운동, 가족, 수면
- 집중이 잘 되는 시간대
- 주말 사용 여부
- 자동화 수준: 제안만, 승인 후 반영

선택:

- 선호 블록 길이
- 회의 후 회복 시간
- 마감 압박을 선호하는지
- 오전형/오후형/야간형

온보딩에서 답하지 않은 정보는 기본값으로 시작하고, 실제 행동으로 보정한다.

### 11.3 명시적 피드백

사용자가 직접 제공하는 신호다.

- 직접 고른 우선순위
- 직접 수정한 소요시간
- 직접 이동한 시간대
- "이 시간 좋음"
- "너무 이른 시간"
- "회의 직후는 싫음"
- "오늘은 못함"
- "이 작업은 중요하지 않음"

이 피드백은 즉시 다음 제안에 반영해야 한다. 사용자가 방금 "회의 직후는 싫음"을 눌렀는데 다음 제안이 또 회의 직후라면 신뢰가 무너진다.

### 11.4 암묵적 행동 학습

사용자가 말하지 않아도 행동에서 배울 수 있는 신호다.

- 승인한 시간대
- 거절한 시간대
- 수정 전후 시간 차이
- 예정 시작 시간과 실제 시작 시간 차이
- 예정 소요시간과 실제 소요시간 차이
- 완료율이 높은 블록 길이
- 반복적으로 미루는 업무 유형
- 회의 많은 날의 수행률
- 마감까지 남은 시간에 따른 수행률

암묵적 학습은 조심스럽게 반영해야 한다. 한 번의 행동으로 성향을 바꾸지 않고, 반복 패턴이 생길 때 가중치를 조정한다.

### 11.5 개인화 프로필

개인화 프로필은 매 요청마다 LLM에 긴 로그를 보내기 위한 것이 아니다. 행동 로그를 주기적으로 요약한 사용자별 운영 규칙이다.

예시:

```json
{
  "preferredFocusWindows": ["09:30-11:30", "15:00-17:00"],
  "avoidWindows": ["13:00-14:00", "immediately_after_meeting"],
  "maxDeepWorkMinutes": 100,
  "defaultTaskDurationMinutes": 45,
  "meetingRecoveryBufferMinutes": 25,
  "weekendScheduling": false,
  "planningStyle": "balanced",
  "automationLevel": "approval_required",
  "taskTypeDurationOverrides": {
    "writing": 90,
    "email": 20,
    "review": 45
  },
  "completionPatterns": {
    "morningDeepWork": 0.82,
    "postMeetingTasks": 0.34,
    "lateEveningTasks": 0.41
  }
}
```

### 11.6 개인화 고도화 단계

1단계: 규칙 기반 개인화

- 온보딩 선호를 그대로 반영한다.
- 사용자가 직접 수정한 우선순위와 소요시간을 저장한다.
- 동일 유형 작업의 평균 소요시간을 반영한다.

2단계: 행동 기반 점수 조정

- 시간대별 승인률과 완료율을 계산한다.
- 회의 전후 성공률을 계산한다.
- 미룸 패턴을 배치 점수에 반영한다.

3단계: 사용자별 예측

- 작업 유형별 예상 소요시간을 개인별로 예측한다.
- 특정 시간대의 완료 확률을 예측한다.
- "이 사용자가 실제로 해낼 가능성이 높은 시간"을 우선 추천한다.

4단계: 제한적 자동화

- 반복적으로 승인된 패턴만 자동 반영한다.
- 위험도가 낮은 작업만 자동화한다.
- 자동화 결과는 언제든 되돌릴 수 있어야 한다.

### 11.7 개인화의 핵심 질문

매 제안은 내부적으로 다음 질문에 답해야 한다.

- 이 일은 정말 오늘 해야 하는가?
- 이 사용자는 이 시간대에 이 유형의 일을 실제로 해내는가?
- 이 일의 예상 소요시간은 이 사용자에게 맞는가?
- 회의, 식사, 휴식, 이동과 충돌하지 않는가?
- 사용자가 직접 말한 우선순위와 AI 판단이 충돌하는가?
- 이번 제안이 이전 실패 패턴을 반복하고 있지 않은가?

## 13. 디자인 방향

### 13.1 톤

제품은 생산성 툴이지만 차갑게 느껴지면 안 된다.

키워드:

- 조용한
- 신뢰감 있는
- 정돈된
- 빠른
- 과하게 꾸미지 않은
- 비서처럼 배려하는

피해야 할 것:

- 마케팅용 과장된 히어로
- 장식 위주의 카드 남발
- AI라는 단어를 과하게 강조
- 사용자가 읽어야 할 설명이 많은 화면

### 13.2 화면 밀도

## 14. 구현 아키텍처 제안

### 14.0 구현 판단

초기 구현은 "빠르게 보이는 데모"보다 "나중에 개인화로 고도화할 수 있는 구조"가 중요하다.

권장 방식:

- 웹앱/PWA로 시작한다.
- Google Calendar 연동을 우선 구현한다.
- LLM은 자연어 이해와 설명에 사용한다.
- 실제 시간 배치는 별도 스케줄링 엔진이 담당한다.
- 사용자 행동 로그는 MVP부터 저장한다.
- 개인화는 사용자별 모델 파인튜닝이 아니라 개인화 프로필과 점수 보정으로 시작한다.

피해야 할 방식:

- LLM에게 "이번 주 일정 짜줘"라고 통째로 맡기는 방식
- 캘린더 이벤트를 직접 수정하는 자동화 우선 방식
- 사용자 행동 로그 없이 화면만 먼저 만드는 방식
- 모든 개인화 로직을 프롬프트 안에 넣는 방식
- 디자인부터 고정하고 데이터 구조를 나중에 맞추는 방식

### 14.1 프론트엔드

- Next.js
- TypeScript
- Tailwind CSS
- 캘린더 UI 라이브러리 또는 커스텀 time grid
- PWA 지원
- 모바일 빠른 입력 최적화
- 데스크톱 주간 플래닝 최적화

### 14.2 백엔드

- Next.js API Routes 또는 별도 Node API
- PostgreSQL
- Prisma
- OAuth token 암호화 저장
- AI 요청 로그와 결과 저장
- 사용자 행동 로그 저장
- 개인화 스냅샷 생성 작업

### 14.3 AI

- 자연어 파싱
- 일정 제안 설명 생성
- 사용자 질문 생성
- 선호도 요약

AI는 구조화 출력이 필수다. 자연어 답변만 받으면 제품 품질이 흔들린다.

AI가 하지 말아야 할 일:

- 캘린더 충돌 계산
- 실제 시간 슬롯 최종 결정
- 승인 없이 이벤트 생성
- 원본 행동 로그 전체를 매번 읽고 판단

AI 출력은 JSON schema로 검증한다. 실패하거나 확신도가 낮으면 사용자에게 질문하거나 기본 규칙으로 fallback한다.

### 14.4 스케줄링 엔진

별도 순수 함수 모듈로 만든다.

책임:

- busy slot 계산
- available slot 계산
- task 후보 배치
- score 계산
- conflict detection
- reschedule proposal
- personalization score 반영

이 모듈은 AI 없이도 테스트 가능해야 한다.

입출력은 순수 데이터로 제한한다.

```text
input:
  events
  tasks
  userPreferences
  personalizationSnapshot

output:
  scheduleSuggestions
  rejectedSlots
  scoreBreakdown
  reasons
```

### 14.5 개인화 엔진

개인화 엔진은 실시간 LLM 호출이 아니라 배치/백그라운드 작업에 가깝다.

역할:

- UserActionLog를 분석한다.
- LearningSignal을 생성한다.
- PersonalizationSnapshot을 업데이트한다.
- 스케줄링 엔진이 사용할 수 있는 간단한 규칙과 점수로 요약한다.

업데이트 주기:

- 중요한 명시적 피드백은 즉시 반영한다.
- 암묵적 행동 패턴은 하루 1회 또는 일정 수 이상의 행동 후 반영한다.
- 신뢰도 낮은 패턴은 사용자에게 노출하지 않고 점수에 약하게만 반영한다.

### 14.6 추천 구현 순서

1. 정적 UX 프로토타입
2. 빠른 입력 저장
3. AI Parser 구조화 출력
4. Google Calendar OAuth와 이벤트 조회
5. FreeBusy 기반 빈 시간 계산
6. 규칙 기반 Scheduling Engine
7. 승인 후 별도 캘린더에 이벤트 생성
8. UserActionLog 저장
9. 완료/미룸/수정 피드백 UI
10. PersonalizationSnapshot 생성
11. 개인화 점수 반영
12. 재배치 자동 제안

### 14.7 기술 리스크와 대응

Google OAuth 검증:

- MVP에서는 개발자 테스트 사용자로 시작한다.
- 공개 서비스 전에는 OAuth consent와 scope 검증을 준비한다.
- 처음부터 최소 권한 원칙을 지킨다.

AI 파싱 오류:

- 구조화 출력 검증을 둔다.
- 확신도 낮은 입력은 질문으로 전환한다.
- 사용자가 수정한 값을 우선한다.

스케줄 품질:

- 충돌 검사는 AI가 아니라 코드로 한다.
- 점수 계산 근거를 저장한다.
- 실패한 제안도 학습 데이터로 남긴다.

개인정보:

- 캘린더 원문 전체를 AI에 보내지 않는다.
- 시간 슬롯과 필요한 제목/분류만 최소 전달한다.
- 토큰은 암호화 저장한다.

성능:

- 캘린더 동기화는 캐시한다.
- 스케줄링 계산은 주간 단위로 제한한다.
- 재배치는 변경된 범위만 다시 계산한다.

## 19. 디자인 요청 프롬프트 초안

Claude 또는 디자인 도구에 전달할 수 있는 프롬프트:

```text
AI Auto Calendar라는 웹/PWA 생산성 서비스를 디자인해줘.

이 제품은 사용자가 떠오르는 일정과 할 일을 Google Keep처럼 빠르게 입력하면, 기존 Google Calendar 일정과 개인 성향을 분석해 실제 가능한 시간대에 타임블록을 제안하고, 사용자가 승인하면 캘린더에 반영하는 AI 비서형 스케줄 오퍼레이터야.

첫 화면은 마케팅 랜딩 페이지가 아니라 실제 앱 화면이어야 해.

핵심 화면:
1. Today 화면: 오늘 일정, AI 타임블록, 지금 해야 할 일, 충돌/위험 알림
2. Quick Capture 입력 영역: 자연어로 막 적을 수 있는 입력창
3. AI 제안 카드: 정제된 제목, 추천 시간, 예상 소요시간, 우선순위, 배치 이유, 승인/수정/미루기 액션
4. Weekly Plan 화면: 주간 캘린더 그리드와 타임블록
5. Preferences 화면: 업무 시간, 집중 시간, 버퍼, 주말 사용 여부, 자동화 수준
6. Learning Feedback: 사용자가 부담 없이 "이 시간 좋음", "너무 짧음", "오늘 못함"을 줄 수 있는 작은 피드백 UI

디자인 톤:
- 조용하고 신뢰감 있는 업무 도구
- 임원 비서가 시간을 정리해주는 느낌
- 과한 AI 브랜딩 없이 실용적
- 정보 밀도는 높지만 복잡해 보이면 안 됨
- 모바일에서도 빠른 입력이 가장 중요함

피해야 할 것:
- 마케팅용 히어로
- 장식적인 카드 남발
- 설명 문구가 많은 화면
- 보라색 AI 서비스 느낌의 과한 그라디언트

사용자가 화면을 보는 순간 "내 하루를 대신 정리해주는 도구"라고 느끼게 해줘.
또한 제품을 며칠 쓰면 "내가 자주 미루는 시간", "실제로 잘 끝내는 시간", "작업별 실제 소요시간"을 조용히 학습해서 제안이 점점 맞아지는 느낌이 들어야 해.
```

## 22. 결론

AI Auto Calendar의 성공 기준은 "AI가 똑똑해 보이는가"가 아니다. 사용자가 하루를 덜 걱정하게 되는가, 그리고 쓸수록 내 생활 패턴에 맞아지는가다.

초기 제품은 다음 약속 하나를 완성해야 한다.

> 아무렇게나 적어도, 내 캘린더와 상황에 맞춰 오늘 할 시간이 잡힌다.

이 약속이 작동하면 제품은 캘린더 앱, 할 일 앱, 메모 앱 사이에 흩어진 사용자의 시간을 하나의 운영 시스템으로 묶을 수 있다.

장기적으로 이 제품은 모든 사용자에게 같은 스케줄을 제안하는 AI가 아니라, 각 사용자의 실제 실행 패턴을 학습하는 개인 비서가 되어야 한다.
